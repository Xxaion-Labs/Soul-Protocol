# DoctrineOS

DoctrineOS is the public direction for a real AI-native operating system built on Doctrine Protocol.

It is not a chatbot, wrapper, plugin, desktop assistant, or agent demo.

DoctrineOS is intended to become a full user-controlled computing environment where AI is part of the operating system itself and Doctrine Protocol supplies the behavioral control layer.

## Core Idea

A normal operating system controls files, apps, permissions, devices, processes, services, and user interaction.

DoctrineOS adds a doctrine-native control spine:

- doctrine files define operating behavior
- concept nodes act like reusable behavior packages
- mount receipts make system behavior inspectable
- adapters connect models, tools, files, apps, and services
- user authority remains the root control law
- powerful actions require bounded, inspectable permission

## Design Direction

DoctrineOS should make the computer feel AI-native without making it autonomous.

The system may help, plan, explain, organize, automate, inspect, and operate tools, but it remains governed by user command and mounted doctrine.

## Public License Direction

DoctrineOS should be public anti-capture infrastructure.

Public software components should use AGPLv3-or-later so public and network-hosted modified versions remain open under the same license family.

## Public/Private Boundary

Public DoctrineOS may include:

- architecture
- runtime code
- public default doctrines
- adapters
- package/registry systems
- receipts and validation
- documentation
- tests

Public DoctrineOS must not include:

- private personal axioms
- private doctrine bodies
- private workflows
- private local-stack internals
- private continuity material
- private identity or life data

Private methods may inform design. They must not be published as source material.
